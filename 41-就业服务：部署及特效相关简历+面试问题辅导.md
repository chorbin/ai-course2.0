## 2. 核心技术栈深度解析

### 2.1 推理框架选择：vLLM vs SGLang vs TensorRT-LLM

2026年主流的三大推理框架各有特点，选择需综合考虑性能、生态、团队能力等因素。

#### 2.1.1 vLLM：通用性与易用性的平衡者

**技术特点**：
- **PagedAttention**：核心创新，将KV Cache管理类比操作系统虚拟内存，极大提升显存利用率
- **连续批处理**：动态调度请求，实现GPU利用率最大化
- **多模态支持**：2026年已稳定支持图像、音频输入
- **生态丰富**：与LangChain、LlamaIndex等主流框架无缝集成

**适用场景**：
```python
# vLLM 典型部署配置
from vllm import LLM, SamplingParams

llm = LLM(
    model="Qwen/Qwen2.5-72B-Instruct",
    tensor_parallel_size=4,        # 4卡张量并行
    gpu_memory_utilization=0.90,   # 显存利用率
    max_model_len=32768,           # 最大上下文长度
    enable_prefix_caching=True,    # 前缀缓存加速
    speculative_model="Qwen/Qwen2.5-72B-Instruct-draft"  # 投机解码
)

sampling_params = SamplingParams(
    temperature=0.7,
    top_p=0.9,
    max_tokens=2048
)
```

**性能数据**（A100 80GB × 4，Qwen2.5-72B）：
- 吞吐量：~4500 tokens/秒（批处理场景）
- 延迟P99：~800ms（单请求2048 tokens输出）
- 显存效率：可支持超过100个并发请求

**优劣势分析**：
| 维度 | 优势 | 劣势 |
|------|------|------|
| 性能 | 高吞吐、低延迟 | 极致优化不如TensorRT-LLM |
| 易用性 | API友好、文档完善 | 高级调优需深入理解 |
| 兼容性 | 支持主流模型格式 | 特定模型可能需适配 |
| 社区 | 活跃、更新快 | 企业支持相对有限 |

#### 2.1.2 SGLang：结构化输出的专家

**技术亮点**：
- **RadixAttention**：基数树前缀缓存，极致复用重复前缀
- **结构化输出优化**：对JSON、代码等格式化输出有专门加速
- **并发编程模型**：基于协程的高效调度
- **Python原生集成**：DSL语法简洁，易于复杂工作流编排

**核心优势示例**：
```python
import sglang as sgl

@sgl.function
def multi_turn_chat(s, question):
    # 结构化输出自动优化
    s += sgl.system("你是一个专业的技术顾问。")
    s += sgl.user(question)
    s += sgl.assistant(sgl.gen("answer", max_tokens=512))
    
    # 前缀缓存自动生效
    s += sgl.user("请详细解释上述回答中的关键点。")
    s += sgl.assistant(sgl.gen("detail", max_tokens=1024))
    
    return s

# RadixAttention自动缓存共享前缀
# 多轮对话场景下可实现3-5倍吞吐提升
```

**典型应用场景**：
- 多轮对话系统（客服机器人、虚拟助手）
- 结构化数据抽取（文档解析、信息提取）
- 代码生成与补全（IDE插件、编程助手）
- RAG系统（检索后生成，大量重复的系统提示词）

**核心优势**：

### 2.3 量化技术与显存优化

**量化策略选择**：

| 方法 | 显存节省 | 性能影响 | 精度损失 | 适用场景 |
|------|----------|----------|----------|----------|
| FP16/BF16 | 50% | 无 | 极小 | 默认选择 |
| INT8 | 75% | 提升10-30% | 可控 | 生产环境首选 |
| INT4-GPTQ | 87.5% | 提升或持平 | 较明显 | 显存紧张场景 |
| INT4-AWQ | 87.5% | 提升或持平 | 小于GPTQ | 平衡精度与性能 |
| FP8 | 75% | 大幅提升 | 极小 | H100+首选 |

**显存计算公式**：
```
模型显存 ≈ 参数量 × 精度字节数 × 1.2（激活+梯度）
KV Cache显存 ≈ 2 × 层数 × 头数 × 头维度 × 序列长度 × 精度字节数 × 批大小

示例：Qwen2.5-72B，INT8量化，4K上下文，批大小32
模型显存 ≈ 72B × 1B × 1.2 = 86GB
KV Cache ≈ 2 × 80 × 64 × 128 × 4096 × 1 × 32 / 1024^3 ≈ 20GB
总需求 ≈ 106GB（4×A100 80GB可承载，留有冗余）
```

## 3. 面试核心问题深度解析

### 3.1 技术原理类问题

#### Q1：请详细解释PagedAttention的工作原理，以及它解决了什么问题？

**考察点**：对vLLM核心机制的理解深度

**标准答案框架**：

**问题背景**：
传统KV Cache管理采用预分配连续内存块的方式，导致两大问题：
1. **内存碎片化**：请求长度不一，固定分配造成大量浪费
2. **内存利用率低**：预分配基于最大长度，实际使用率常低于30%

**PagedAttention解决方案**：
借鉴操作系统虚拟内存管理，将KV Cache分割为固定大小的"页"（通常为16或32 tokens）：
- **按需分配**：根据实际生成长度动态分配页
- **非连续存储**：页可以不连续，通过页表映射
- **内存共享**：不同请求可共享相同的前缀页（如系统提示词）

**具体机制**：
```
Block Table（页表）:
Request 1: [Block 0] → [Block 3] → [Block 7] → ...
Request 2: [Block 0] → [Block 5] → [Block 9] → ...
                    ↑
                共享前缀（如system prompt）
```

**性能提升**：
- 显存利用率从30%提升到95%+
- 支持更大的批处理，吞吐提升2-4倍
- 相同硬件可支持更多并发请求

**进阶追问**：
- 如果请求大量共享前缀，如何进一步优化？（答案：结合SGLang的RadixAttention）
- PagedAttention的额外开销是什么？（答案：页表管理、间接内存访问）

#### Q2：请解释连续批处理（Continuous Batching）与传统批处理的区别？

**考察点**：对推理调度机制的理解

**标准答案**：

**传统批处理**：
```
批次开始 → 等待所有请求完成 → 批次结束
问题：短请求需等待长请求，GPU空转
```

**连续批处理**：
```
时刻T: [Req1(parsing), Req2(generating), Req3(generating)]
时刻T+1: [Req2(generating), Req3(generating), Req4(parsing)]  ← Req1完成，Req4加入
时刻T+2: [Req3(generating), Req4(generating), Req5(parsing)]
```

**核心机制**：
1. **动态插队**：新请求无需等待整个批次完成
2. **即时退出**：完成的请求立即释放资源
3. **迭代级调度**：每个生成步骤都重新评估批次组成

**代码示例**：
```python
# 伪代码展示连续批处理逻辑
class ContinuousBatchScheduler:
    def __init__(self, max_batch_size):
        self.max_batch_size = max_batch_size
        self.running_requests = []
        self.waiting_queue = Queue()
    
    def step(self):
        # 移除已完成的请求
        self.running_requests = [
            req for req in self.running_requests 
            if not req.is_finished()
        ]
        
        # 填充新请求
        while (len(self.running_requests) < self.max_batch_size 
               and not self.waiting_queue.empty()):
            new_req = self.waiting_queue.get()
            self.running_requests.append(new_req)
        
        # 执行一步推理
        return self.run_inference_step(self.running_requests)
```

**性能对比**：
- 传统批处理：GPU利用率40-60%
- 连续批处理：GPU利用率80-95%

### 3.4 项目经验类问题

#### Q5：请描述一个你负责的最具挑战性的部署项目，包括问题、方案和成果？

**考察点**：实战经验、问题解决能力、成果量化

**优秀回答模板**：

**背景**：
> 某头部电商公司在双十一期间需要支撑智能客服系统，要求QPS从日常的500峰值提升到5000，同时P99延迟不超过500ms，且预算有限，只能使用现有的GPU资源池（20张A100）。

**挑战**：
1. 流量峰谷比达10:1，传统静态部署资源浪费严重
2. 客服场景长尾问题多，需要长上下文（16K），显存压力大
3. 多租户隔离，不同商户有独立模型微调版本需求
4. 7×24高可用，不允许停机

**方案设计**：

```
阶段一：基线优化（提升30%吞吐）
├── 切换到vLLM，启用连续批处理
├── INT8量化，显存占用减半
└── 优化CUDA内核，减少CPU-GPU同步

阶段二：架构改造（支撑峰值流量）
├── 实现模型热加载，支持多版本共存
├── 开发智能调度器，短对话优先
├── Redis缓存热点问答，命中率35%
└── 开发流量预测模块，提前10分钟扩容

阶段三：成本优化（降低30%运营成本）
├── 混合精度推理（FP16+INT8混合）
├── 空闲时段释放GPU实例
└── Spot实例处理低优先级请求
```

**技术亮点**：
```python
# 智能调度器核心逻辑
class SmartScheduler:
    def predict_qps(self, historical_data):
        # 基于历史数据预测未来10分钟QPS
        model = self.load_model('qps_predictor.onnx')
        return model.predict(historical_data[-60:])  # 最近60分钟数据
    
    def auto_scale(self, predicted_qps):
        current_instances = len(self.active_instances)
        required = predicted_qps / self.qps_per_instance
        
        if required > current_instances + 2:
            # 预扩容
            self.scale_up(int(required) - current_instances)
        elif required < current_instances - 1:
            # 预缩容
            self.scale_down(current_instances - int(required))
```

**成果量化**：
- 峰值QPS达到5200，超额完成目标
- P99延迟控制在420ms
- GPU利用率从平均35%提升到78%
- 运营成本降低28%
- 故障率从0.5%降低到0.02%

**经验总结**：
> 这个项目让我深刻理解到，AI部署不是简单的技术堆砌，而是需要在性能、成本、可靠性之间找到最佳平衡。关键在于建立可观测性体系，让系统"可度量、可预测、可控制"。
